Adaptive Client

Build a scalable, loosely coupled service layer across multiple platforms and transports.

What Adaptive Client does

Adaptive Client is a library that allows a developer to write services that target multiple transports and database platforms. Each implementation is registered with the dependency injection container using a connection string or URL as a root key. An entire dependency graph can be resolved effortlessly at runtime, dependent on external factors such as configuration setting, Internet availability, or availability of a specific server.

For example, the developer of a desktop application might implement a configuration setting that allows the end user to choose either MSSQL or MySQL as a database platform. The developer may also implement a configuration option that will route data calls to an Internet API server instead of directly to a DBMS. The application itself does not need any special code to handle settings such as these. Adaptive Client handles the task of looking at these configuration settings. It resolves and injects the correct service layer components at runtime and the application runs correctly.

Get the source code on GitHub. Adaptive Client is free and open source, even for commercial use.

Registering services with a dependency injection container is a complex job. It often involves using factory methods to inject lower level dependencies and other parameters. When multiple dependency graphs are necessary, managing registrations becomes truly burdensome. Adaptive Client eliminates much of the work by providing registration wrapper methods that are easy to understand, implement, and maintain. Adaptive Client also provides special registration methods for Entity Framework. These methods manage the lifetime of the DbContext object and allow multiple asynchronous calls to be made in parallel.

Adaptive Client introduces an object called a Service Manifest. A Service Manifest is simply a directory of services that are made available by the service layer. It is a single object with each service exposed as a property. This single object is easily injected into a Controller or ViewModel. It provides an immediate gateway to the entire service layer. Guidance to each service and each service method is provided by Visual Studio Intellisense.

Get the Adaptive Client nuget package.

Adaptive Client makes it easy to write services that are granular and testable. In large applications, the service layer often becomes a monolith. This usually happens because services are highly inter-dependent. When there are hundreds of services, the dependency graph becomes unmanageable and developers resort to creating services in a single, enormous class. Adaptive Client resolves this problem by making the Service Manifest available to each service. A service can lazily resolve an instance of any other service and no additional effort is required by the developer.

Adaptive Client is not a service locater. It is strongly typed throughout.

Common use cases for Adaptive Client

Perhaps the most common use case for Adaptive Client is thick-client applications that are run on corporate local area networks. These types of applications often have their service layer wrapped by some transport such as REST, WCF, or gRPC. This is done to allow data to be exposed to consumers who need access to the data via the Internet.

The problem in this scenario is that LAN users, who often represent the vast majority, are reduced to making out-of-process HTTP calls to access data. These users have LAN connectivity, which, if utilized, would allow them to make in-process database calls that are hundreds of times faster. Adaptive Client rectifies this by allowing different implementations of the application service layer to be injected into the application at run time. An implementation for LAN users would make fast in-process database calls. A different implementation for Internet users would make calls over HTTP to an API server.

Another common use case is that of web servers that reside on the same physical local area network as their API counterparts. These servers are often configured to make HTTP calls when they need data. A much more efficient way to obtain the data would be to bypass the API server and make an in-process call directly to the database. Adaptive Client allows different implementations of the service layer to be resolved at run time which would allow a web server to access data in the most efficient way possible.

A third use case scenario is that of an application that is designed to target multiple database platforms. The application developers will of course need to write a correct implementation of the service layer for each DBMS they choose to target. The application, however, need not have any additional logic implemented as result of a decision to target multiple database platforms. Adaptive Client will be responsible for looking at configuration settings, resolving the dependency graph, and injecting service components into the application.

How Adaptive Client works

Much of the functionality in Adaptive Client is centered around a class called End Point Configuration. This class has four important string properties:

  • Connection String:  Database connection string or API URL.
  • API Name:  User defined name that describes the database or API (OurApp, Accounting, BankAPI).
  • End Point Type:  User defined description of the end point type (REST, WCF, gRPC, InProcess).
  • Provider Name: User defined name of a database provider (MSSQL, MySQL, MemSQL, None).

An End Point Configuration is created for every connection string or API URL. Every class in the service layer is registered with the dependency injection container using one or more properties of an End Point Configuration as a key. When a dependency graph needs to be created, Adaptive Client is able resolve the correct components based on the connection string being used.

The current version Adaptive Client works only with Autofac. Support for other dependency injection containers will be added in the future.
